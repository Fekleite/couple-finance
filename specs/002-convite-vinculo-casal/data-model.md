# Data Model: F02 - Convite e vinculo do casal

## Shared Budget

**Table**: `public.shared_budgets`

Representa o espaco financeiro compartilhado do casal. Nesta feature ele e um
container vazio para features futuras.

### Fields

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | `uuid` | Yes | Primary key, generated by database |
| `created_by_user_id` | `uuid` | Yes | References `auth.users(id)` |
| `name` | `text` | Yes | Default like `Espaco compartilhado`; no financial values |
| `status` | `text` | Yes | `active` or `archived`; F02 creates only `active` |
| `created_at` | `timestamptz` | Yes | Defaults to `now()` |
| `updated_at` | `timestamptz` | Yes | Updated on changes |

### Validation and Constraints

- `status` must be one of `active`, `archived`.
- Creator must also have an active creator membership.
- Future financial records must reference `shared_budgets.id`, not invitation
  rows.

### RLS

- Members can select only budgets where an active `budget_members` row exists
  for `(select auth.uid())`.
- Authenticated users can create a budget only through the planned create flow
  or RPC that also creates the creator membership.
- No anonymous access.

## Budget Member

**Table**: `public.budget_members`

Representa a participacao de um usuario em um orcamento compartilhado.

### Fields

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | `uuid` | Yes | Primary key |
| `shared_budget_id` | `uuid` | Yes | References `shared_budgets(id)` |
| `user_id` | `uuid` | Yes | References `auth.users(id)` |
| `role` | `text` | Yes | `creator` or `partner` |
| `status` | `text` | Yes | `active` or `removed`; F02 creates active only |
| `created_at` | `timestamptz` | Yes | Defaults to `now()` |
| `updated_at` | `timestamptz` | Yes | Updated on changes |

### Validation and Constraints

- `role` must be one of `creator`, `partner`.
- `status` must be one of `active`, `removed`.
- Unique active membership per user:
  `unique (user_id) where status = 'active'`.
- Unique active membership per budget/user:
  `unique (shared_budget_id, user_id) where status = 'active'`.
- Maximum two active members per budget must be enforced by trigger or RPC
  guard because partial unique indexes cannot express count limits directly.
- Exactly one active creator per active budget should be enforced by partial
  unique index on `(shared_budget_id)` where `role = 'creator' and status =
  'active'`.

### RLS

- Active members can select member rows for their own budget.
- Users can select their own membership rows.
- Insert/update of partner membership during invitation acceptance should happen
  through the transactional accept RPC or guarded database logic.
- No anonymous access.

## Invitation

**Table**: `public.budget_invitations`

Representa o pedido para outra pessoa entrar no orcamento compartilhado.

### Fields

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | `uuid` | Yes | Primary key; can be used in invitation URL |
| `shared_budget_id` | `uuid` | Yes | References `shared_budgets(id)` |
| `inviter_user_id` | `uuid` | Yes | References `auth.users(id)` |
| `invitee_email` | `text` | Yes | Display value after trim |
| `invitee_email_normalized` | `text` | Yes | Lowercase trim; used for lookup |
| `status` | `text` | Yes | `pending`, `accepted`, `declined`, `cancelled`, `expired` |
| `expires_at` | `timestamptz` | Yes | `created_at + interval '7 days'` |
| `responded_at` | `timestamptz` | No | Set on accepted/declined/cancelled/expired transition |
| `created_at` | `timestamptz` | Yes | Defaults to `now()` |
| `updated_at` | `timestamptz` | Yes | Updated on changes |

### Validation and Constraints

- `status` must be one of `pending`, `accepted`, `declined`, `cancelled`,
  `expired`.
- `invitee_email_normalized` must be lowercase trimmed and non-empty.
- Inviter cannot invite their own authenticated e-mail.
- Only one pending invitation per `shared_budget_id`.
- Optional defensive unique index can prevent duplicate pending invite for the
  same normalized e-mail.
- Pending invitation is valid only while `status = 'pending'` and `expires_at >
  now()`.
- Terminal states are `accepted`, `declined`, `cancelled`, `expired`.

### RLS

- Inviter can select pending and terminal invitations they sent.
- Authenticated invitee can select invitations where normalized auth e-mail
  matches `invitee_email_normalized`.
- Inviter can cancel only their own pending invitations.
- Invitee can decline only pending invitations addressed to their authenticated
  e-mail.
- Acceptance should use a transactional RPC that validates the same rules and
  inserts the active partner membership.
- Unrelated authenticated users and anonymous users cannot see invitation
  details.

## Relationship State

**Source**: Derived from `budget_members` and `budget_invitations`.

### States

| State | Meaning | Primary action |
|-------|---------|----------------|
| `loading` | Session or relationship query in progress | No private data shown |
| `no_shared_budget` | No active membership and no relevant pending invitation | Create shared space and invite |
| `invitation_sent` | User is creator/member with pending invitation sent | Cancel invitation or wait |
| `invitation_received` | User has pending invitation addressed to auth e-mail | Accept or decline |
| `couple_linked` | User has active membership in shared budget | Show shared context active |
| `invitation_unavailable` | Link is invalid, expired, terminal, unauthorized or blocked | Show safe unavailable message |
| `error` | Recoverable service/network failure | Retry without exposing private data |

### State Priority

1. If session is not authenticated, ProtectedRoute controls access.
2. If relationship query is loading, show loading state.
3. If user has active membership and a pending sent invitation, show
   `invitation_sent`.
4. If user has active membership and two active members exist, show
   `couple_linked`.
5. If user has pending received invitation and no active membership, show
   `invitation_received`.
6. If no membership or invitation exists, show `no_shared_budget`.
7. If invitation URL cannot be used safely, show `invitation_unavailable`.

## State Transitions

| Trigger | From | To | Notes |
|---------|------|----|-------|
| Create shared budget and invite | `no_shared_budget` | `invitation_sent` | Creates budget, creator member and pending invitation |
| Accept invitation | `invitation_received` | `couple_linked` | RPC inserts partner member and marks invitation accepted |
| Decline invitation | `invitation_received` | `no_shared_budget` | Marks invitation declined; no member created |
| Cancel invitation | `invitation_sent` | `no_shared_budget` or linked creator alone | Marks invitation cancelled; no partner member created |
| Expire invitation | `pending` invitation | `invitation_unavailable` or no pending state | Can be derived by `expires_at <= now()` and optionally materialized |
| Retry terminal invitation | terminal state | unchanged | Idempotent safe message |
